Smart Expense Agent — Stage Summary

Working document · 2026-05-13 · Based on code review of lib/ and architecture discussions to date.

1. Executive Summary

Smart Expense Agent is a Flutter mobile app that lets Israeli company employees photograph one receipt — or a tray of up to ~20 in a single shot — and have Gemini 2.5 Flash automatically extract every financial field (invoice number, date, business, final amount after discounts, VAT, parking entry/exit times, line items, project mapping, and category in Hebrew). The user verifies the results on a review screen and confirms.

This document consolidates the decisions made so far across four areas: backend architecture, data storage, backend language, and unit economics / pricing on the app stores.

2. Backend Architecture — Server-Authoritative, Not WhatsApp-Style

Decision

Data is stored server-side as the source of truth, with a local SQLite cache on the device for fast UI and offline queueing. The "WhatsApp local-only" model was explicitly rejected because expense data is corporate financial data, not personal.

ConstraintWhy local-only fails
Israeli tax retention — 7 years of receipts (תקנות מס הכנסה — ניהול פנקסים)Lost phone or uninstall = lost audit trail.
Multi-user workflow (employee scans → manager approves → accountant exports)No shared state without a server.
Continuity (employees leave, get new devices)Data must outlive any single device.
Cross-employee dedup and project-list distributionRequires a central source of truth.

Why a backend exists at all (three non-negotiables)

  1. Hide GEMINI_API_KEY — today the key is bundled into the app via .env as a Flutter asset, which leaks it inside the APK/IPA. This must move server-side before any external release.
  2. Enforce multi-tenant isolation by companyId claim in Firebase Auth custom claims.
  3. Be the system of record for the 7-year retention requirement.

3. Storage Strategy — DB ≠ Image Store

Metadata (the parsed Receipt JSON) and binary images live in different services. Firestore is not used for binary blobs; it has a hard 1 MiB document limit and is not designed for that workload.

AssetWhere it livesWhy
Parsed receipt JSON (invoice #, amount, items, etc.)Firestore documentQueryable, indexed, real-time listeners.
Original receipt image (compressed JPEG ~150–300 KB)Cloud Storage (Standard tier first 30 days → Coldline)Cheap, durable, signed-URL access.
AI-detected cropsCloud Storage alongside originalsOptional but useful for quick thumbnails.
Image hash (SHA-256)Firestore (alongside the imageRef)Tamper-evidence for audit; also enables cheap re-scan caching.
Do originals need to be kept at all? Yes — if the app is to serve as the company's system of record for tax purposes, the Israeli Tax Authority's rules on "digital scanning" require retention of the scanned image for 7 years (otherwise the paper original must still be kept in a folder). Storage cost is negligible (see §6), so the default recommendation is to store originals.

4. Database — Recommendation: Firestore

OptionVerdictWhy
FirestorePrimary recommendationNative integration with Firebase Auth + Cloud Storage + Gemini (all Google Cloud). Real-time listeners. Tenant isolation via security rules. Zero ops.
Supabase (Postgres)Strong alternativeReal SQL, joins, Hebrew full-text search, self-hostable. Slightly more setup.
Self-hosted PostgresSkip for nowOperational overhead not justified at this stage.

Firestore data model (top-level collections)

companies/{companyId}                       ← name, defaultVatRate, settings
companies/{companyId}/projects/{id}         ← name, address (used for project mapping)
companies/{companyId}/users/{uid}           ← role: employee | manager | admin
companies/{companyId}/receipts/{id}         ← parsed Receipt JSON + imageRef + imageHash
companies/{companyId}/scans/{scanId}        ← raw Gemini response, for audit / re-run

OptionVerdictWhy

TypeScript + Cloud Functions (Gen 2)Fine for v1Simpler deploy; slightly more cold-start variance.

API Surface (small on purpose)

POST /v1/scan          → multipart image → Gemini → JSON      (the proxy)
POST /v1/receipts      → bulk write after user "Confirm all"
GET  /v1/receipts      → paginated history
GET  /v1/projects      → projects for the user's company
POST /v1/export/xlsx   → server-rendered Excel for accountants

Auth: Firebase ID token in Authorization: Bearer …. The backend extracts companyId from custom claims — clients never pass it.

Cost levers (when scale demands it)

  1. Batch better — pack ~10 receipts per Gemini call. Cuts AI cost ~5×.
  2. Cache Gemini responses by imageHash — re-scans of the same image cost $0.
  3. Tier images to Archive ($0.0012/GB) after 1 year → another 3× reduction on storage.
  4. Two-pass model — Flash-Lite for detection, full Flash for OCR (future optimization).

Pricing to net $2 / employee / month

Let P = monthly price charged per employee. After 15% store cut: net revenue = 0.85 × P. Subtracting variable cost ($0.03) and solving for $2 net profit: 

0.85 × P − $0.03 = $2
→ P = $2.39 / employee / month

Recommendation: list at $4.99 / employee / month

  • Competitive landscape sits at $4–$18 (Expensify, Zoho Expense, SAP Concur). $2.49 signals "toy product" to corporate buyers.
  • The Hebrew-receipts moat (categories, VAT, supermarket discount handling) is defensible — charge for it.
  • $4.99 still undercuts every major competitor in the Israeli market while netting $4.21 per seat after the store cut.
  • Suggested go-to-market hook: first 3 seats free, then $4.99/seat.

Heavy-user caveat

A road warrior scanning 200 receipts/month costs ~$0.18 in cloud + AI; at $4.99 list this still nets $4.06. Add a fair-use clause for > 500 receipts/month and route those companies to an Enterprise plan.
